## Design Context

### Users

Detour is a Chromium extension for web developers and designers doing local testing — redirecting HTTP requests to a local server, injecting scripts into third-party pages, stripping CSP/CORS/X-Frame-Options headers. The surface area is small (a popup and a devtools panel) but the job it does is load-bearing: one wrong toggle and someone's morning is gone.

The primary context is fast: someone clicks the toolbar icon, scans their rules, flips one on, closes the popup, reloads the tab. A secondary context is slower — first-time setup or editing the regex of a rule that's misbehaving.

The core job: **know at a glance what's active on this page, and change it with one click.** Everything else (pattern syntax, DNR vs. fetch-patching, service-worker caching, regex capture groups) is plumbing the user shouldn't have to see unless they go looking.

### Brand Personality

**Calm, confident, crafted.** Detour should feel like it was made by designers, for designers — not a developer afterthought with a UI slapped on.

Emotional goal: *"This was made for me, not for developers."*

Three words: **calm, confident, crafted.**

### Aesthetic Direction

- **References:** Medium (typography, whitespace), Linear (polish, attention to detail), Notion (simplicity, approachability).
- **Anti-references:** Azure DevOps, Jira, Chrome DevTools chrome, anything with a dark ops-dashboard aesthetic, rounded-corner cards with emoji icons in colored circles.
- **Typography:** Serif headlines (Newsreader), clean sans body (Instrument Sans). Editorial/magazine feel. Mono (JetBrains Mono / SF Mono) is reserved for *identifiers* — rule names, URLs, code — not for chrome.
- **Layout:** Left-aligned everything. No center-aligned text blocks. Generous whitespace. The popup is a narrow column (~360–420px), so hierarchy must come from type, not horizontal bands.
- **Theme:** Both light and dark, following system preference (`color-scheme: light dark`).

#### Iconography

- **Model:** Quiet, utility-first, visually restrained.
- **Role:** Icons are not the personality of the UI. Typography, spacing, and layout carry the tone; icons should behave like calm signage.
- **Style:** Stroke-first, low-detail, optically balanced shapes on a 16px or 20px grid, with consistent weight, rounded joins/caps, and generous negative space.
- **Semantics:** One icon communicates one idea. Prefer small, repeatable metaphors over bespoke illustrations.
- **State:** Active and selected states come from the container, spacing, or pill/background treatment — not from making the icon itself louder.
- **Color:** Neutral by default. Color only for true semantic meaning: success, warning, error.
- **Avoid:** Tiny "app window" drawings, decorative fills, mixed icon styles, overloaded metaphors, or icons trying to explain an entire feature.
- **Bias:** Between a quiet generic icon and a clever expressive one, prefer the quieter.

#### Color Palette

```
Teal:     #2A7598  — primary accent, CTA buttons, links (light)
Teal-lt:  #6CB6D6  — primary accent (dark mode)
Green:    #2EA043  — success (dark: #73C991)
Red:      #D03A3A  — errors (dark: #F48771)
Amber:    #8A5D00  — warnings, in-progress (dark: #E5BA7D)
```

Teal is the only chrome color. The others are semantic and should appear only when they mean something.

#### Spacing Scale

4px-based grid, exposed as CSS custom properties `--sp-1` through `--sp-13`. Use tokens for all padding, margin, and gap. Raw pixel values are acceptable only for sub-4px optical adjustments (1–2px badge padding, inline code nudges).

```
--sp-1:  4px    --sp-5: 12px    --sp-9:  24px    --sp-13: 48px
--sp-2:  6px    --sp-6: 14px    --sp-10: 28px
--sp-3:  8px    --sp-7: 16px    --sp-11: 32px
--sp-4: 10px    --sp-8: 20px    --sp-12: 40px
```

### Design Principles

1. **Left-aligned, editorial layout.** Center alignment signals "developer tool." Left-aligned with generous measure signals "crafted for reading."
2. **Hide the plumbing.** DNR, content scripts, MAIN-world injection, regex anchors, service-worker caches — all invisible by default. The user sees: the URL being scoped, rule name, source → destination, on/off. Everything else is tucked behind disclosure or shown only when an error demands it.
3. **Real state, not fake.** Every indicator reflects actual extension state. A green badge means a rule actually fired; a toggle being on means rules are actually registered. No animated spinners pretending work is happening.
4. **One click for the common case.** Returning users resume with one click (flip a rule, open the popup). First-timers need two clicks max to a working rule.
5. **Typography carries the hierarchy.** Use size, weight, and font contrast (serif vs. sans vs. mono) instead of color or decoration. Serif for warm moments (empty-state title), mono for identifiers (URLs, rule names), sans for UI chrome and controls.
